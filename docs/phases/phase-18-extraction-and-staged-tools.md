# Phase 18 — extraction subscription tier + staged-write/control tools (§8 Phase 2; P1.2–4,12; FP-4)

Two coupled streams, disjoint files: (A) wire the 3 **extraction** roles through the phase-16 router + the subscription **single-tier** machinery; (B) the **staged-write + control** MCP tools. After this an interactive Claude can drive the whole learning loop on the subscription — **manually, through the same gates.**

**Prime directive:** ADD, never REMOVE. **DEFAULT==TODAY** (extraction stays local+cloud two-tier until the runner is enabled+healthy). Staged writes are the ONLY agent write path; the human-gated spine (§5) is NEVER exposed.

## Stream A — extraction subscription tier (`agents/extraction/*`, `triggers/{sessionEnd,jobs}.ts`)
- **Route the 3 roles through the router** (per phase-16 integration notes, deferred here): `extraction.tiebreak` (`resolve.ts` — `router.forRole('extraction.tiebreak', ctx.jobId)` as the `ExtractionLlm`); `extraction.verify` (`verify.ts` — `router.complete('extraction.verify', {…, taskId: ctx.jobId})`, keep the §17 self-judging guard); `extraction.fuzzy` (`fuzzy.ts` local passes ← `router.forRole('extraction.fuzzy', ctx.jobId)`). `router?` is optional on `ExtractionAgentDeps`; absent → today's `llm`/`cloud` byte-identical.
- **The escalation-mode branch** (the subtle part, per the phase-16 integration notes): decide once per run from `router.resolve('extraction.fuzzy').backend`. `subscription-claude` → **single tier** (Gate A/B become no-ops; chunk at `EXTRACTION_SUBSCRIPTION_CHUNK_TOKENS` 30k, cap `EXTRACTION_SUBSCRIPTION_PASS_MAX_TOKENS` 2k — the phase-14 constants; P1.4). `local-qwen3`/`cloud-api` → today's two-tier Gate A/B, unchanged. Default resolves local → two-tier runs; the new branch is dormant until opt-in.
- **New tier/pass values:** `ExtractionTier += 'subscription'` (types.ts); `ExtractionPass += 'llm-subscription'` (types.ts — stamps `extraction@0.0.1/llm-subscription`); `write.ts` `tierPass` learns `'subscription' → 'llm-subscription'`.
- **`skipped-subscription-extractor`** (verify.ts, P1.3): when `extraction.tier === 'subscription'`, mirror the `skipped-cloud-extractor` guard (below-gate items → human review) UNLESS an independent `cloud-api` verifier is configured (§2.2 recommended). Add the mode to the `VerifyState['mode']` union.
- **The `delegate` step + continuation handler** (P1.2) — a delegate-consuming extraction variant: loads `runner_submissions` for a task into a `FuzzyExtractionState` (tier `'subscription'`), **re-chunks the transcript locally** with `chunkTranscript(text, EXTRACTION_CLOUD_CHUNK_TOKENS, estimatingTokenCounter())` to populate `chunkTexts` (P1.2 — **identical chunking to `read_session`** so the child's `chunk` indices line up; add a test pinning index equivalence), then `resolve → verify → write` unchanged. Checkpoint after loading so a crash never re-buys. This variant powers the interactive `submit_extraction_items` continuation NOW and agent-mode extraction in phase-19.
- **`SessionEndOrigin += 'mcp'`** (sessionEnd.ts) for `run_extraction`. **Continuation-task family** (P1.12): id `extract-cont-<sessionId>-<sha8(batch)>`, kind `'extraction'`, payload `{ continuation: true, sessionId }` → routed to the delegate variant; **exempt `extract-cont-*` from `runTaskRetentionSweep`'s "kept forever" clause** (they are NOT exactly-once tokens — let them sweep after retention).

## Stream B — staged-write + control tools (`mcp/tools/{write,control}.ts`, `skills/{agent,candidate}.ts`)
Fill the phase-14b registry seams. Every tool rides the chokepoint + is already tiered in permissions (STAGING_TOOLS / CONTROL_TOOLS).
- **`propose_extraction`** (STAGED-WRITE) — payload IS the `ExtractionPayload` `commitExtraction` validates (`security/stagedWrites.ts`) → `INSERT INTO staged_writes (kind 'extraction', …)`; provenance stamped server-side (`extraction@0.0.1/llm-subscription`, `proposed_by='claude-mcp:<sid>'`). Reply = the `propose_correction` shape.
- **`submit_extraction_items`** (STAGED-WRITE) — components/preferences/corrections (zod, ≤60/kind), normalized via `normalizeComponent/Preference/Correction` (export them from `fuzzy.ts` if needed), `INSERT OR IGNORE INTO runner_submissions` (deterministic ids `sha256(taskId+kind+normalized text)`). When NOT in a bound runner task, **synthesize a continuation task** (Stream A's `extract-cont-<sid>-<sha8>`, `{continuation:true}`) so interactive Claude's submissions get processed.
- **`propose_skill_revision`** (STAGED-WRITE) — validates the SKILL.md at the boundary (`parseSkillMd`, frontmatter name == baseline, differs from active; `hasPendingReview` → `INVALID_STATE`), then `queue.enqueue({kind:'skill-improvement', priority: TASK_CLASS_BAND.user + TASK_PRIORITY.skillImprove, payload:{skillId, providedCandidate:{versionId, instructions, proposedBy}}})`. **It does NOT skip the gate** — the candidate is benchmarked on the held-out split and adopts only through the §17 gate.
- **Control tools:** `run_extraction` (`enqueueExtraction(queue, {sessionId,…}, 'mcp')`), `improve_skill_now` (`enqueueManualImprovement`), `run_maintenance` (`prune`/`export` fire), `retry_task` (`queue.retryDeferred` from 14a — map `TaskRetryError` code → `NOT_FOUND`/`INVALID_STATE`), `scan_watched_folder` (`scanWatchedFolder`).
- **Skills `providedCandidate` branch** (`skills/agent.ts` candidate step + `candidate.ts`): when `state.providedCandidate` is present, validate + use it (recompute the version id via `candidateVersionIdOf`) instead of calling the rewrite LLM — **before** the no-cloud guard (a provided candidate needs no cloud). `testset`/`benchmark`/`write` unchanged.

## NEVER exposed (§5 — the human-gated spine)
No `approve`/`reject`/`decide`/`undo`/`grant`/`registerAgent`/`setApiKey`/`revealMcpToken`/`installHook`/`watch.add`/`watch.remove`/`skills.rollback`/`skills.improvementSettings`. Reads open; writes only as staged proposals; the decision stays human.
