# Phase 09 report — Security: sandbox lanes, permissions, audit/undo, staged writes

**Status:** done · **Date:** 2026-07-04

## What was built

### `src/main/security/` — the §13 safety spine (10 modules)

- **`capabilities.ts`** — the capability schema, THE single source for both lanes and the kernel gates. `CapabilityDeclarationSchema` (zod): `{ fsRead[], fsWrite[], netDomains[], tools[], maxSpendUSD }`, every omitted field defaulting to deny (empty/0). `parseCapabilities` normalizes untrusted declarations (`~` expansion, absolute-path requirement — a declaration must never depend on the daemon's cwd — dedupe, lowercased hosts); the dependency-free `CapabilityDeclaration` *interface* lives in `kernel/types.ts` (it is the §13 contract) with a compile-time pin that the zod output satisfies it. Matching helpers shared by every consumer: `isPathWithin`/`pathsAllowed` (win32 case-folded, `..`-traversal-safe), `isDomainAllowed` (Deno `--allow-net` grammar: bare host = any port, `host:port` = that port, no subdomain wildcards), `capabilitiesWithin` (sandbox requests must not exceed the agent's own declaration). Lane derivations (§11 "a near 1:1 mapping"): `denoPermissionFlags` (no capability ⇒ no flag ⇒ Deno's default-deny; `--no-prompt` — deny instead of interactive hang — and `--no-remote` — rule code is local, remote imports would be an undeclared network path — are unconditional) and `dockerCapabilityArgs` (`--network none`, `--cap-drop ALL`, `--security-opt no-new-privileges`, `--read-only` root + tmpfs `/tmp`, fsRead[i] → `/caps/read/<i>:ro`, fsWrite[i] → `/caps/write/<i>:rw`).
- **`permissions.ts`** — `PermissionEngine implements PermissionChecker`, replacing the phase-04 kernel stub. DEFAULT-DENY: unregistered agents are hard-blocked on everything, including reads. Registered agents pass the §13 tiered gates: **auto-allow** safe reads/retrieval (`storage-read`, `retrieval`, `model-call`, `workflow-step`, in-scope `fs-read`); **approval-gated** writes/net/spend/side-effecting sandbox runs — without a standing grant the action inserts a `pending` row in the new `approvals` table and is NOT executed ("headless = stays queued"); an approval/denial persists per action *signature* (sha256 of agent+kind+name+facts), so the dashboard decision lets the retried action through (or keeps it blocked) durably; **hard block** for out-of-scope facts (path outside fsRead/fsWrite, host not in netDomains, undeclared tool, spend above `maxSpendUSD`, sandbox requesting beyond the agent's declaration) — no approval row, span event via the kernel. Agent registry supports exact ids and prefix families (`mcp:` → one profile per transport-session family, longest prefix wins). `registerInternalAgents` installs the boot profile (decision 3).
- **`deno.ts`** — the §11 default lane. `ensureDenoBinary`: the MANAGED binary — Deno v2.9.1 pinned by version + per-platform zip sha256 in config.ts (digests from the official GitHub release API, re-verified against the downloaded archive), resumable download (`.part` + HTTP Range, the phase-02 reranker pattern), sha256 verified BEFORE extraction, atomic write + rename, in-flight promise dedup. `DenoLane implements SandboxLane`: lazy ensure; spawns `deno run --quiet <derived flags> --v8-flags=--max-old-space-size=<mb> <entry>`; download/spawn failures are structured `spawn-failed` results, never throws.
- **`docker.ts`** — the §11 polyglot lane. `detectDocker` (detect-and-guide per §1: `docker version` probe, 10 s hung-CLI kill, cached per process, actionable install/start guidance). `DockerLane`: capability args derived FIRST (fail-closed before daemon contact), `docker run --rm -i --name sbx-<hex> --memory/--memory-swap --pids-limit 128 --cpus 1 <capability args> -v <entryDir>:/sandbox:ro alpine:3.22 sh /sandbox/<entry>`; on timeout the container is killed BY NAME (killing the docker CLI kills only the client).
- **`sandbox.ts`** — the shared lane contract: `SandboxRunRequest`/`SandboxResult`/`SandboxLane` and `collectSandboxProcess`, the plumbing both lanes share so the JSON stdin/stdout contract, output bounds (1 MiB), stderr tails, and the wall-clock SIGKILL cannot drift between them.
- **`zip.ts`** — dependency-free ZIP extractor for the single-file Deno archives (EOCD scan, central-directory walk, stored + deflate via `node:zlib`; zip64 and multi-disk throw clearly; CRC32 deliberately skipped — the outer archive is sha256-pinned before extraction).
- **`audit.ts`** — the §13 audit + undo log (appdata `audit_log`). `graphWrite(agentId, description, fn)` wraps ONE write-lane job (§21 rule 1) with a recording `WriteTx`: each structured op captures its inverse BEFORE executing — created node → `delete-node`, updated node → `restore-props` with the touched properties' pre-images (read inside the lane job; Dates → ISO; null = was-unset), newly created edge → `delete-edge`, re-MERGEd existing edge → no inverse. Raw *mutating* cypher inside an audited job has no generic inverse ⇒ the whole action is flagged un-undoable (never guessed); reads don't poison reversibility. A job that throws mid-way still logs (outcome `error`) with the inverses captured so far — lane jobs are exclusive, not transactional, so the committed prefix is real and *rollback-able from the dashboard*. `fileWrite`/`fileDelete` copy pre-images into `backups/audit/<actionId>/` first. `undo(actionId)`: graph inverses apply newest-first in ONE lane job (`restore-props` goes through `upsertNode` so TIMESTAMP decode + the HNSW embedding re-index dance are handled); file undo restores the pre-image or removes the created file; the undo itself is audited (`kind='undo'`, un-undoable — no redo in v1); `NOT_FOUND`/`IRREVERSIBLE`/`ALREADY_UNDONE`/`UNDO_FAILED` are typed errors. Implements the phase-04 `AuditHook` seam, so every kernel-mediated action now lands as a durable `kind='action'` row.
- **`stagedWrites.ts`** — the §13 lifecycle: propose → human-readable diff → approve (commit via lane, audited) / reject. Handles BOTH proposer payload shapes: `propose_correction` (`{patch, reason}`, protected keys re-refused at commit — defense in depth against out-of-band row edits) and `extraction` (self-contained `{op, node, embedOnCommit, edges, tagCreates, provenance, evidence, reason, session}`; labels/edge types validated against the §18 registry, never trusted). `renderStagedWriteDiff`: corrections show `~ prop: current → proposed` against the LIVE node; extraction items show `+ CREATE`/`~ MERGE`, props, planned edges, tag creates, provenance and evidence. `approveStagedWrite`: status `staged → approved` (decision recorded) `→ committed` (ONE audited lane job; `embedOnCommit` embeds the statement with the shared bge-m3 at commit time — embeddings are never staged); a commit failure leaves the row `approved` with the error in `validation_json` and approve() retries. `rejectStagedWrite`: the row's own status is the ONLY effect — no trace beyond the log (DoD-pinned: zero lane jobs).
- **`untrusted.ts`** — `UntrustedText`, the type-level §21-rule-5 containment: ingested/tool/document content is wrapped at its boundary and travels as a class that is NOT assignable to `string`, so it can never reach a tool-call constructor, `KernelAction`, sandbox entry path or Cypher statement (pinned at compile time by `@ts-expect-error` tests). Coercion traps: `toString`/`Symbol.toPrimitive`/`toJSON` yield a redacted marker — accidental interpolation into a query/log/payload leaks nothing. The ONLY unwraps are two named sinks: `untrustedForStorage` (content becoming inert stored data) and `untrustedForPromptData` (content embedded in a model prompt AS DATA).
- **`scanner.ts`** — the §13 detection layer: regex + LLM instruction-pattern scanner at ingest. Eight named regex patterns for the canonical shapes (override-instructions, exfiltrate-to-url, imperative-tool-call, reveal-hidden-prompt, secrets-exfiltration, chat-template-tokens, decode-and-execute, act-as-override) run always, offline, deterministic; the local qwen3 judges regex-clean documents with a schema-constrained verdict (`{"suspicious": bool, "reason": string}` — the phase-08 structured-outputs finding applied), consulted only when a model is configured; model failures degrade to regex-only with a warning (§13: detection is the fallible layer; containment and undo are the reliable ones). A flag NEVER blocks ingestion — content is stored as inert data regardless; findings return in the ingest result AND persist to appdata `injection_flags` for the phase-10 dashboard.

### Kernel replacement (`src/main/kernel/`)

`Kernel` now REQUIRES an injected `PermissionChecker` (the phase-04 stub is gone; `allowAllPermissions()` exists as an explicit test stand-in so the safety spine is visible at every construction site). Every action — allowed, queued or blocked — produces a span whose attributes carry `permission.decision` (`allow`/`pending`/`block`) + `permission.reason` (+ `permission.approval_id`); denied/queued actions throw INSIDE the span, so blocks land in the traces table with status `error` — the §13 "hard block + span event". ## Key decisions & findings (read before later phases)

1. **The §13 capability schema has no "graph" field on purpose** — `fsRead`/`fsWrite` are host filesystem paths, `netDomains` is network, `tools` is the tool manager, `maxSpendUSD` is cloud spend. The graph is the shared store agents coordinate through (§13 "share only via the store"), and its protections are its own §13 machinery: the single write lane, the staged-writes review flow, provenance, and now per-action undo. Kernel `storage-write` actions still classify into the write tier for gating.
2. **Standing grants are how the tiered gates coexist with phases 03–08.** "Prompt before writes/net/spend" cannot literally block a live MCP session (§15 forbids pause-and-notify) or the extraction agent (whose writes already pass §17's confidence gate + independent verifier + staged review — an approval row would be an approval of an approval). So gated tiers consult the agent profile: `'allow'` = standing consent recorded at registration, `'ask'` (the DEFAULT for every tier) = pending-approval row. Internal registrations (recorded here, § "internal agents"): `system` (write grant for internal maintenance jobs), `extraction-agent` (write + spend grants, spend capped at the §20 $0.50 ceiling — the same figure the SpendMeter enforces per call), `mcp:` prefix (write grant; read tools + `propose_correction` are auto-allow tiers anyway — staging IS the §21-rule-6 gate). User rules in phase 11 get NO standing grants: their write/net/spend actions queue for the dashboard.
3. **Approvals persist per action signature** (sha256 of agent + kind + name + scope facts). That is what makes "headless = stays queued" workable: the denied-but-queued action can be retried verbatim after the dashboard approves, and an identical future action inherits the decision. Distinct facts (a different path/host/amount) form a new signature and queue fresh.
4. **The Docker lane fails closed on `netDomains`** (measured constraint, pinned by conformance case 7): plain `docker run` cannot enforce per-domain egress — `--network` is all-or-none, DNS pinning is bypassable by raw IPs, and an in-container iptables sidecar needs privileges we refuse to grant. A declaration this lane cannot honor is `refused` (a denial), never silently granted broader access. Network-capable rules belong in the Deno lane, whose `--allow-net=<domains>` enforces the allowlist for real. Revisit only with a proxy-sidecar design (phase 13+, if polyglot rules ever need egress).
5. **Conformance asserts OUTCOMES, not mechanisms.** Deno denies with `NotCapable`; Docker denies because an undeclared path was never mounted and the network doesn't exist. Both are "the code could not do it", which is what the capability promised. Corollary (documented in capabilities.ts): whether a *write* grant implies *read* access on the same path is lane-divergent (Deno: no; a Docker rw mount: yes) — declare `fsRead` for anything a rule reads; the conformance table only promises what both lanes deliver identically.
6. **CPU-time caps are approximated by the wall-clock kill** — Node exposes no cross-platform per-process CPU rlimit, so a spinning sandbox burns at most `timeoutMs` of wall clock (SIGKILL; Docker containers are additionally killed BY NAME since killing the CLI kills only the client, plus `--cpus 1` and `--pids-limit 128`). Memory is real in both lanes (`--v8-flags=--max-old-space-size` / `--memory` + `--memory-swap`).
7. **Audited writes capture inverses per STRUCTURED op; raw mutating cypher makes the action un-undoable** — no generic inverse exists for arbitrary Cypher, and guessing one would corrupt the graph on undo. Consequence visible today: fresh-create knowledge ingests are fully reversible; replace-jobs (which `DETACH DELETE` old chunks via raw cypher) are honestly flagged un-undoable — their recovery story is re-ingest (content-hash pipelines are self-healing). Undo restores content, not timestamps: `updated_at` restamps on restore (engine-owned).
8. **A failed audited job logs the inverses captured so far** — lane jobs are exclusive, not transactional (§5), so a mid-job crash leaves a committed prefix; the audit row (outcome `error`) can roll that prefix back from the dashboard. Pinned by test.
9. **`UntrustedText` is a class, not a branded string** — non-assignability to `string` is the entire defense, and it must survive accidental `${}`/`JSON.stringify` (coercion traps return a redacted marker). The wrapper rides the ingestion pipeline end-to-end (boundary wraps in file read / MCP inline / docstring digest; the single unwrap is the storage sink in knowledge.ts). The extraction transcript deliberately stays string-typed INSIDE the workflow: its state is JSON-checkpointed (a class doesn't survive serialization), and its §21-rule-5 protections are behavioral and pinned by phase-08 tests (tool_result bodies never reach prompts; extracted content lands behind the gated write). Recorded interpretation of "end-to-end" per spec §0.
10. **The scanner flags, never blocks** (§13's own framing: detection is the fallible layer; containment + undo are the reliable ones). Regex runs always and is deterministic — the DoD phrase is caught offline with zero model calls; the LLM pass covers regex-clean phrasing (schema-constrained qwen3 verdict — the phase-08 structured-outputs finding transfers; probed 3/3 stable) and degrades to regex-only on failure. False positives cost a dashboard row, false negatives cost nothing new (the content was already inert data) — so patterns lean toward recall. `[^\n]` bounds the verb→URL gap (dots appear in paths like `~/.ssh`), found while testing the DoD fixture.
11. **`propose_correction`'s protected keys are re-checked at commit time** — the staging tool refuses them, but a `staged_writes` row edited out-of-band (or a future second proposer bug) must not rewrite identity/provenance through an approval. Same defense-in-depth: extraction payload labels/edge types re-validate against the §18 registry before any op runs.
12. **`Kernel.permissions` is REQUIRED, `allowAllPermissions()` is explicit** — a default-allow fallback would silently reintroduce the phase-04 stub; making every construction site name its checker keeps the safety spine visible. All pre-existing integration suites now run the real engine via the shared fixture (internal registrations + `agent-demo`), which is how the phase proves phases 03–08 still work UNDER the gates, not around them.
13. **Deno pins**: v2.9.1 (released 2026-07-01), per-platform zips + sha256 from the release API, `win32-x64`/`darwin-x64`/`darwin-arm64`/`linux-x64`. The managed download lands in `<binDir>/deno-v<version>/` so version bumps are side-by-side; the extracted exe is not re-hashed per spawn (it can only exist via a pin-verified zip; hashing ~99 MB per run would defeat the lane's millisecond-startup purpose).
14. **`.gitattributes` pins LF for `tests/fixtures/sandbox-probes/`** — the repo checks out with `core.autocrlf=true` and CRLF breaks busybox `sh`; the conformance suite additionally normalizes on copy (probes are byte-copied to a temp dir off OneDrive — files-on-demand can break Docker bind mounts).
