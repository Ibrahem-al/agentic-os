# Phase 04 report — Kernel: workflow runner, context manager, tracing

**Status:** done · **Date:** 2026-07-04

## What was built

### `src/main/telemetry/` — OpenTelemetry → SQLite (§9 observability)

- **`exporter.ts`** — `SqliteSpanExporter implements SpanExporter`, writing the phase-01 `traces` table in appdata.db (SQLite, not the graph — span writes never contend with the single write lane). Row mapping: trace/span/parent ids, name, kind, start/end unix-ms, status (`ok`/`error`/`unset`), attributes JSON (+ `otel.status_message` for errors). Paired with a `SimpleSpanProcessor` and better-sqlite3's synchronous writes, a span's row is **committed by the time `span.end()` returns** — spans from a process that is later SIGKILLed are already durable (only never-ended spans are lost, inherent to tracing).
- **`tracer.ts`** — `createTelemetry(db) → Telemetry`: a `BasicTracerProvider` per instance (deliberately NOT the OTel global, so test instances stay isolated) + the process-wide `AsyncLocalStorageContextManager` (registered once) so `withSpan` nesting rides ambient context through any async code. `withSpan(name, attrs, fn, {parent?})` = start → run in context → status OK/ERROR + recordException → always end. `remoteParentContext(traceId, spanId)` builds a remote-parent context — the resume mechanism: spans started under it join a trace begun in an earlier (possibly dead) process.

### `src/main/kernel/` — the §9 mediation layer

- **`types.ts`** — the thin interfaces agent code sees: `WorkflowStep` (plain step list; `run(state, ctx) → patch` shallow-merged into state), `WorkflowRunner` (`define`/`run`/`resume`/`getJob`), `ActionExecutor` (the §13 chokepoint shape), `KernelAction`/`PermissionDecision`/`AuditHook`, `SummarizerLlm` (structural, satisfied by `OllamaClient`). **Nothing here imports LangGraph.**
- **`checkpointer.ts`** — `SqliteCheckpointSaver extends BaseCheckpointSaver` over appdata.db tables `workflow_checkpoints` + `workflow_checkpoint_writes` (appdata schema v2, additive; `openAppData` upgrades v1 in place and still refuses newer versions). Written in-house instead of installing `@langchain/langgraph-checkpoint-sqlite` so checkpoints live in OUR appdata.db through the dual-ABI better-sqlite3 handle; semantics mirror the upstream MemorySaver/SQLite savers: `put` (serde-serialized checkpoint+metadata, parent id from the incoming config), `putWrites` (special channels ERROR/INTERRUPT/… map to negative idx and REPLACE; regular writes INSERT OR IGNORE), `getTuple` (exact or latest — checkpoint ids are uuid6, lexicographically time-ordered — plus pendingWrites and parentConfig), `list` (desc, `limit`/`before`/metadata `filter`), `deleteThread`. Born on checkpoint format v4 with empty tables, so upstream's pre-v4 migration path is deliberately absent.
- **`runner.ts`** — `LangGraphRunner implements WorkflowRunner`; **the only module that imports LangGraph** (ESLint-enforced, see below). `define(name, steps)` validates (non-empty, unique, non-reserved step names; no redefinition) and lazily compiles a linear `StateGraph` whose single `data` channel holds the whole JSON state (reducer = shallow merge). Every step executes through the injected `ActionExecutor` → `kernel.workflow-step` span + PHASE-09 permission seam, with `WorkflowStepContext` (jobId/workflow/step/index) passed to the step. `run(name, input, {jobId?, agentId?})` inserts the job into the §8 **`tasks` table** (a workflow job IS a task: `kind='workflow'`, payload = workflow/agentId/input/trace ids), opens a `workflow.run` root span, **persists the root span's trace/span ids into the payload BEFORE invoking** (so a killed process leaves enough behind to continue the trace), and invokes with **`durability: 'sync'`** — the post-step checkpoint is committed before the next step starts, which is what makes kill-mid-step resume work. Success → task `done`, returns jobId; step failure → task `failed` + `last_error` + rejects `WorkflowJobError` (retry/backoff policy stays with the phase-11 scheduler per §15). `resume(jobId)` re-reads the job, requires the definition (definitions are code; only state is durable), bumps attempts, opens `workflow.resume` **parented on the original run's remote span context** (same trace id across processes), and invokes with `null` input = continue from the latest checkpoint. Crash-left `running` rows resume fine; `done` jobs resume as a no-op. `getJob` returns status/attempts/state/nextSteps (graph snapshot when defined; raw checkpoint read otherwise).
- **`context.ts`** — the §10 context manager. `ContextManager.assemble({objective, sections[], system?, provider?, tokenBudget?, reserveOutputTokens?})` builds a prompt within the active provider's budget (default: `PROVIDER_CONTEXT_WINDOW_TOKENS[provider] − reserve`). Sections fit **verbatim or summarized — never truncated, never dropped**: pinned scaffolding (system + objective + headers) and pinned sections must fit or assembly throws loudly; non-pinned sections waterfill (sections fitting an equal share stay verbatim, their slack redistributes; the rest split the remainder as summary targets, each below `CONTEXT_MIN_SUMMARY_TOKENS` → throw). Summarization is map-reduce on the LOCAL tier (§10): chunk to `CONTEXT_SUMMARIZE_CHUNK_TOKENS` (sized for Ollama's 4096 default num_ctx), per-chunk `generate` with a **hard output cap** (`num_predict` = share of target × `CONTEXT_SUMMARY_OUTPUT_FRACTION` — the estimator's overestimate margin is why the cap sits below target), re-summarize up to `CONTEXT_SUMMARIZE_MAX_ROUNDS`, then throw if non-convergent (only possible if the model ignores its cap repeatedly). The summarizer prompt demands **facts first** (identifiers/codenames/numbers preserved exactly) precisely because output caps truncate tails — front-loading makes cap-truncation harmless. Each chunk call is a `model.summarize` span. Token counting stays the phase-03 per-provider **estimating** counter (imported from `retrieval/tokens`, which is dependency-free) — see decision 6.
- **`kernel.ts`** — `Kernel implements ActionExecutor`: `execute(agentId, action, fn)` = permission check (`// PHASE-09` pass-through stub returning allowed; the throw-on-deny path and `KernelPermissionError` are the contract the real §13 engine fills in) → `kernel.<kind>` span with agent/action attributes → audit record (`// PHASE-09` in-memory stub `createAuditLogStub()`; the real reversible-delta audit log replaces it) with outcome ok/error + duration. Every action already creates a span.
- **`index.ts`** — barrel; the rest of the app imports only from here.

## Key decisions & findings (read before later phases)

1. **Checkpointer is in-house** (`SqliteCheckpointSaver`), not `@langchain/langgraph-checkpoint-sqlite`: keeps checkpoints in OUR appdata.db through the dual-ABI better-sqlite3 loader, zero extra native deps, and the upstream contract is small and pinned by tests. If upstream's checkpoint format moves past v4, revisit (`serde` + `copyCheckpoint` come from `@langchain/langgraph-checkpoint`, so serialization tracks the pinned version).
2. **`durability: 'sync'` on every invoke** — LangGraph 1.x defaults to 'async' (checkpoint persisted while the next step runs), which would race a SIGKILL. 'sync' + better-sqlite3's synchronous writes make "checkpoint durable before next step starts" a hard guarantee; the kill test depends on it.
3. **Jobs live in the §8 `tasks` table** (`kind='workflow'`) — a workflow job IS a task; the phase-11 scheduler reads the same table and can pre-create ids via `RunWorkflowOptions.jobId`. A crashed process leaves `status='running'`; `resume()` deliberately accepts that state. `run()`/`resume()` resolve the jobId at completion and reject with `WorkflowJobError` on step failure — retry/backoff/defer is scheduler policy (§15), not runner policy.
4. **Trace continuity across processes** is data, not magic: the run root span's (traceId, spanId) persist in the job payload before step 1 executes; `resume()` rebuilds a remote parent context from them. A killed process's root span never lands in `traces` (spans export on end) — the trace still knits together through the ids, and the DoD test pins the exact shape.
5. **Steps route through `ActionExecutor`** (the Kernel), so "span per workflow step" and "span per kernel action" are the same span (`kernel.workflow-step`, action name = step name) and PHASE-09's permission check will automatically cover every step of every background agent with zero runner changes.
6. **Per-provider tokenizers stay estimators** (phase-03 finding 10 stands): no §20 provider ships a local tokenizer (Anthropic = counting API only; tiktoken et al. would be off-pin deps). The conservative estimating counter is the §10 counting layer; its deliberate overestimate is load-bearing — it's why summary output caps sit below targets (`CONTEXT_SUMMARY_OUTPUT_FRACTION`) and why an assembled prompt can't blow a real window. Swap point if real tokenizers ever land: `retrieval/tokens.ts` behind `TokenCounter`.
7. **Rule-12 picks** (recorded; none are §20 values): provider windows 200k/200k/1M/128k (conservative floors — a small window only makes summarization kick in earlier), local window 4096 (Ollama default num_ctx; qwen3:4b itself accepts 262k), summarize chunk 2048 est-tokens, max 3 rounds, output fraction 0.6, min honest summary 32 tokens; appdata `user_version` 2; job payload shape; span table mapping (events not persisted — attributes + status only; add a column later if the phase-10 trace panel wants events).
8. **qwen3:4b as a summarizer — probed limitations (matters for phase 08):** with `think:false` this quant NARRATES ("We are given a list…") instead of answering directly, for summarize-style tasks at any temperature; with `think:true` the hidden thinking consumes the ENTIRE `num_predict` budget (probe: 1,680 tokens of thinking, empty response) — never combine think:true with tight output caps; Ollama `format:` (JSON-schema constrained decoding) stops the narration but the model then echoes the instructions as the "summary" (vacuous). What DOES work: the facts-first system prompt reliably front-loads concrete identifiers/codenames into the first ~100 output tokens even inside narration, so facts survive capped summaries on realistic varied content (live test pinned at that operating point). Highly repetitive content can still defeat it (the model fixates on counting repetitions). Summaries from the local tier should be treated as fact-carriers, not clean prose; the §20 small LLM is user-swappable in settings for users who want better.
9. **`SimpleSpanProcessor` over `BatchSpanProcessor`** on purpose: batch would lose ended-but-unflushed spans on a kill, and the "high-frequency" concern §9 raises is already solved by SQLite being lock-free from the graph lane's perspective. If span volume ever hurts (phase 11 watchers), switch to batch + `forceFlush` on checkpoint boundaries.
10. **OTel globals**: only the context manager (AsyncLocalStorage) is registered process-wide — that's mechanics, not state. The tracer provider is per-`createTelemetry` instance, so parallel test databases don't cross-write spans (pinned by test).
11. **ESLint boundary layout changed**: flat-config rule objects REPLACE (not merge) `no-restricted-imports` for overlapping file sets, so the config now has explicit blocks per zone (everything / storage / kernel / tests). If a later phase adds another driver boundary (e.g. MCP SDK), extend those blocks rather than adding an overlapping one.
12. **Windows quirk**: `@langchain/langgraph` + OTel packages are pure JS — no `rebuild:native` changes. The kill test bundles its child with esbuild `packages:'external'` into `out/test-child/` (gitignored) so node resolves real node_modules; it runs under plain node and under `ELECTRON_RUN_AS_NODE` (dual-ABI resolver picks the right better-sqlite3 in the child too).

## Deferred / notes

- `kernel.execute` is not yet called by anything except workflow steps (no agents exist until phase 08); the facade + audit stub are the seams PHASE-09 fills in. `KernelPermissionError`'s deny path is dead code by design.
- Context manager `provider: 'local'` prompts (assembling for the local tier itself) — not needed until a local background agent exists; the summarizer already respects the local window via chunking.
- Span events/links are not persisted (attributes + status only). Revisit with the phase-10 trace panel if the waterfall wants event dots.
- `getDeltaChannelHistory` (beta upstream API) uses the inherited default implementation — fine at our scale; override in the checkpointer if graphs ever use DeltaChannels heavily.
- The §8 scheduler heap/aging/priorities are phase 11; the runner deliberately exposes exactly the hooks it needs (caller-supplied jobId, status transitions in `tasks`, `WorkflowJobError` for retry policy).

- Wrap every MCP tool call in `kernel.execute(sessionAgentId, {kind: 'mcp-call', name: toolName, …}, fn)` — you get the span and the PHASE-09 permission seam for free; also log to `mcp_calls` per §12 (that table is separate from tracing).

- `@modelcontextprotocol/sdk` is §20-pinned but NOT yet installed — install it exact-pinned in phase 05.
